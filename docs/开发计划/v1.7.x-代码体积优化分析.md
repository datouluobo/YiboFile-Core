## 2. 深度分析与重构建议

### 2.1 TrainingWindow.xaml.cs (3750+ 行)

**问题分析**:
- **上帝类 (God Class)**: 这个文件几乎包含了所有东西——UI 事件处理、模型训练流程控制、数据库交互、标签自动补全逻辑、图片加载逻辑等。
- **UI 与逻辑强耦合**: 大量的业务逻辑直接写在事件处理函数(`_Click`, `_SelectionChanged`)中，难以单元测试。
- **可维护性差**: 修改任何一个小功能都需要在 3000 多行代码中定位，极易引入回归 Bug。

**重构建议**:
1.  **引入 MVVM 模式**: 将业务状态和逻辑移至 `TrainingWindowViewModel`。
2.  **拆分功能模块**:
    - `ITrainingSessionManager`: 负责训练流程控制（开始、暂停、下一张）。
    - `ITagSuggestionService`: 负责标签自动补全和预测逻辑。
    - `IImageLoaderService`: 负责图片加载和缓存。
3.  **使用 Partial Class (短期方案)**: 如果 MVVM 改造成本过高，至少按功能将文件拆分为多个 `partial class` (e.g., `TrainingWindow.Tagging.cs`, `TrainingWindow.Training.cs`)。

### 2.2 DocumentPreview.cs (2160+ 行)

**问题分析**:
- **巨大的 Switch/If-Else**: 推测该类包含大量针对不同文件后缀的硬编码判断逻辑。
- **职责单一原则违背**: 同时处理了 PDF, Office, Text, Code 等多种格式的预览生成。

**重构建议**:
1.  **策略模式 (Strategy Pattern)**: 定义 `IPreviewHandler` 接口，为每种（或每类）文件格式实现单独的 Handler (e.g., `PdfPreviewHandler`, `OfficePreviewHandler`)。
2.  **注册机制**: 使用工厂模式或依赖注入容器来根据文件扩展名获取对应的 Handler，消除巨大的 Switch 语句。

### 2.3 TabService.cs (1930+ 行)

**问题分析**:
- **UI 逻辑泄露**: Service 层本应只处理数据 logic，但包含了大量关于 Tab 宽度计算 (`MeasureTextWidth`)、UI 元素操作的代码。
- **混合职责**: 同时管理 Tab 的生命周期（增删改查）和 UI 表现。

**重构建议**:
1.  **分离 UI 控制器**: 将 UI 相关的计算逻辑提取到 `TabUiController` 或 `TabLayoutManager` 中。
2.  **纯化 Service**: `TabService` 只负责维护 `ObservableCollection<TabItemModel>` 数据源，不直接操作 UI 控件。

### 2.4 Settings Panels (多个 ~1200 行文件)

**问题分析**:
- 所有设置项的 `Load`, `Save`, `UI Change` 逻辑都堆在一个文件中。

**重构建议**:
1.  **分层管理**: 将设置数据的读写与 UI 分离（如果尚未分离）。
2.  **组件化**: 将复杂的设置组（如“高级设置”）提取为独立的 UserControl。
